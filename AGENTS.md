# Repository Guidelines
## Project Structure & Module Organization
The CodeIgniter 4 core lives under `app/` (Controllers, Models, Views, Config, helpers). Module-specific front ends such as `appraisal/`, `recruitment/`, `stats/`, and `demo1/` hold legacy UI bundles and assets; keep new PHP controllers in `app/Controllers` and surface public files through `public/`. Shared static assets (images, scripts, styles) belong in `public/` and should mirror module names. Runtime logs, cache, and uploads stay confined to `writable/`. Automated tests and fixtures are organised beneath `tests/` (`unit`, `database`, `session`, `_support`). Vendor libraries are managed by Composer in `vendor/` and must remain untouched.
## Build, Test, and Development Commands
Run `composer install` after cloning or pulling to sync dependencies. Launch the app locally with `php spark serve --port 8080`, which proxies requests to `public/index.php`. Execute `composer test` (alias for `phpunit`) for the entire suite, or `phpunit --filter ModuleNameTest` when iterating. Regenerate class maps with `composer dump-autoload` after adding namespaces, and use `php spark migrate` to apply new migrations.
## Coding Style & Naming Conventions
Adhere to PSR-12 with four-space indentation and trailing newlines. Name controllers `SomethingController`, services under `App\Services`, and view files with snake_case reflecting route intent (e.g., `attendance_overview.php`). Method names use camelCase; database columns remain snake_case to match schema. Keep configuration changes isolated under `app/Config`, and prefer environment keys over hard-coded credentials. Lint before committing with `php -l path/to/file`.
## Testing Guidelines
Place unit tests in `tests/unit` mirroring application namespaces and name classes `{Subject}Test`. Database-dependent specs belong in `tests/database` with seeds defined in `Tests\Support\Database\Seeds`. Use descriptive method names like `test_handles_empty_schedule()` and cover both success and failure paths. Refresh migrations in a disposable database via `php spark migrate:refresh --seed` before running integration tests; note any manual QA steps in the PR.
## Commit & Pull Request Guidelines
Craft imperative, 72-character commit subjects (`Add payroll export job`) with brief bodies listing rationale and coverage. Group related edits and avoid mixing refactors with behaviour changes. Pull requests should summarise scope, reference tracker IDs, list executed commands (`composer test`, manual flows), and attach screenshots or API traces for UI or HTTP work. Tag module owners (`appraisal`, `recruitment`, `stats`) for review and call out schema or public API changes in a dedicated section.
## Security & Configuration Tips
Copy `env` to `.env` locally and keep secrets out of source control. Configure `app.baseURL`, database credentials, and mail transports per environment. Restrict write permissions on `writable/` and never expose it via the web server. Sanitize and validate all uploads before storage, and rotate encryption keys whenever provisioning a new tenant or migrating environments.
